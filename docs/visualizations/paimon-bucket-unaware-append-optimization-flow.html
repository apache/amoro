Paimon BUCKET_UNAWARE APPEND 表优化流程

该动画按规划后的职责拆分展示：AMS 负责扫描发现、规划、任务调度和最终提交；Spark 优化器只接收已规划的 AppendCompactTask 执行文件合并，并返回 CommitMessage。

AMS: 规划 + 调度 + 提交 Spark: 只执行合并 Paimon: 快照 + 文件存储
初始化
准备加载调度链路。
扫描事件
AMS 规划与提交侧
OptimizingQueue / TableRuntime
周期调度表运行时，读取待优化分区与资源配额，决定是否进入 Paimon 优化流程。
ProcessFactoryRouter
只有配置启用 Paimon process factory 后才路由到 Paimon；默认关闭避免误接管。
PaimonAppendFileScanner
以 snapshot 作为扫描基线，读取 APPEND 表文件元数据，按 partition 收集 ADD 文件。
PaimonOptimizingPlanner
评估分区健康度，生成 MINOR / MAJOR / FULL 计划，并固定 commitUser 与 commitIdentifier。
TaskFilesPersistence
把 PaimonCompactionInput 持久化，内容包含已序列化的 AppendCompactTask。
OptimizingService 调度
按可用 core 下发任务，任务状态从 PENDING 进入 RUNNING，失败可重试或触发重规划。
PaimonTableCommit
AMS 聚合 CommitMessage，使用 table.newCommit(commitUser).filterAndCommit(...) 提交。
Spark 优化执行侧
SparkOptimizingTaskFunction
从 AMS 拉取任务输入，反序列化后交给 executor factory；不重新扫描 Paimon 表。
PaimonCompactionExecutor
只调用 AppendCompactTask#doCompact 合并文件，禁止本地规划和本地提交。
PaimonCompactionOutput
返回序列化 CommitMessage、输入/输出文件数量和资源耗时指标。
执行边界
Spark 不调用 scan/plan/commit；不处理 clusterIncrementalEnabled、dataEvolution、aware-bucket 或 PK 表。
Paimon 表与快照
APPEND 快照与 Manifest
扫描阶段读取当前 snapshot 的 file metas，targetSnapshotId 只用于漂移判断。
小文件集合
BUCKET_UNAWARE 表没有固定 bucket 约束，规划以 partition 文件集合为主。
p=20260513 / 18 files p=20260514 / 11 files p=20260515 / 7 files
合并后的数据文件
Spark 输出较大的 compacted data files，但先不改变表快照。
COMPACT Snapshot
AMS commit 成功后才生成新快照；同一 commitIdentifier 避免重复提交。
固定排除范围
仅支持 Paimon 1.3.1 的 APPEND + BUCKET_UNAWARE + OrderType.NONE。

当前阶段说明

    任务调度队列

    AMS 按资源配额下发任务，Spark 完成后只回传结果，最终提交仍由 AMS 串行聚合。
    T-1partition=20260513PENDING
    T-2partition=20260514PENDING
    T-3partition=20260515PENDING

    职责边界

    AMS扫描 snapshot、评估分区、生成任务、持久化输入、调度资源、聚合输出、提交快照。
    Spark执行已规划的 AppendCompactTask，产生 CommitMessage，不做计划和提交。
    Paimon提供 1.3.1 内核 API、文件元数据和 commit 语义，最终由 AMS 驱动提交。
